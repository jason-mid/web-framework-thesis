# Flask Analysis\n\n## Routing\nFlask uses the `@app.route` decorator to map URL patterns to view functions. Routes can include variable parts (e.g., `<int:id>`) and support HTTP methods via the `methods` argument. The routing system is built on Werkzeug’s URL matching, providing flexibility and easy customization.\n\n## Templating\nFlask integrates Jinja2 as its default template engine. Auto‑escaping is enabled for HTML‑related extensions (`.html`, `.htm`, `.xml`, `.xhtml`, `.svg`). Global objects such as `config`, `request`, `session`, `session`, `session`, `url_for`, `url`, `url`, `url etc., are automatically available in templates.\n\n## WSGI Compliance\nFlask is a lightweight WSGI application. An instance of `flask.Flask` is a WSGI callable that can be run under any WSGI server (Gunicorn, uWSGI, etc.).\n\n## Extension Ecosystem\nFlask’s functionality can be extended through a large ecosystem of extensions (`Flask‑SQLAlchemy`, `Flask‑Login`, `Flask‑Migrate`, etc.). Extensions follow a simple pattern: they are instantiated, then initialized with the app via `init_app(app)`. The official docs list how to find, install, and build extensions.\n\n---\n\nThis file provides a concise reference for the thesis background on Flask.
